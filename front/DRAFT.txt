1. Аутентификация и Пользователь (Supabase Auth + Express)
Сама регистрация и логин происходят на фронте (Next.js) через SDK Supabase. Но бэкенду нужно знать, что пользователь существует.

POST /api/users/webhook

Зачем: Секретный эндпоинт. Когда пользователь успешно регистрируется через Supabase на клиенте, Supabase посылает сюда вебхук, чтобы ты создал запись в своей таблице users (с настройками валюты, часовым поясом).

GET /api/users/me

Зачем: Получить профиль пользователя и его настройки (например, предпочитаемую валюту для дашборда).

PATCH /api/users/me

Зачем: Обновить настройки (сменить валюту, включить/выключить email-уведомления).

2. Управление подписками (Core CRUD)
Здесь обязательно используй валидацию входящих данных (например, Zod), чтобы в БД не попадал мусор.

GET /api/subscriptions

Зачем: Получить список всех активных подписок пользователя. Поддерживает query-параметры для сортировки (например, ?sortBy=next_payment_date).

POST /api/subscriptions

Зачем: Добавить новую подписку вручную.

Body: name, amount, currency, billing_cycle (monthly/yearly), first_payment_date, category.

Логика: Бэкенд сам высчитывает и сохраняет поле next_payment_date на основе даты первой оплаты и цикла.

GET /api/subscriptions/:id

Зачем: Получить детали конкретной подписки (включая ссылку на инструкцию по ее отмене).

PUT /api/subscriptions/:id

Зачем: Отредактировать подписку (изменилась цена или дата).

DELETE /api/subscriptions/:id

Зачем: Удалить (или перевести в статус "cancelled"), если пользователь отменил сервис.

3. Дашборд и Аналитика
Фронтенду нужны агрегированные данные для красивых графиков, чтобы не считать это на клиенте.

GET /api/analytics/summary

Зачем: Возвращает общую сводку.

Response: Общая сумма трат в месяц, сумма в год (burn rate), список подписок, которые спишутся в ближайшие 7 дней.

GET /api/analytics/chart

Зачем: Возвращает данные, подготовленные для графиков (например, траты по категориям: Entertainment, Work, Utilities).

4. "Умные" фичи (Киллер-фичи для резюме)
POST /api/import/csv

Зачем: Пользователь загружает CSV-файл банковской выписки.

Логика: Express парсит файл, находит повторяющиеся транзакции (через простую логику или обращение к OpenAI API) и возвращает массив предполагаемых подписок. Пользователь на фронте подтверждает их, и они улетают в POST /api/subscriptions.

5. Планировщик и Уведомления (Cron)
Этот роут не вызывается из браузера. Его дергает внешний планировщик (например, cron-job.org, GitHub Actions или Supabase Cron) раз в сутки.

POST /api/cron/notify-upcoming

Зачем: Сервер проходит по БД: SELECT * FROM subscriptions WHERE next_payment_date = CURRENT_DATE + 3.

Логика: Для каждой найденной записи отправляет письмо через сервис вроде Resend или SendGrid. Обязательно защити этот роут секретным ключом в заголовках, чтобы его нельзя было дернуть снаружи.
